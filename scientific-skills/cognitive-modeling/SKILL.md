# Cognitive Modeling

## Purpose

This skill provides expert guidance for selecting and applying computational cognitive models across multiple frameworks. Computational cognitive modeling decomposes observed behavior into latent cognitive processes using formal mathematical models. A competent programmer without cognitive science training would not know which modeling framework is appropriate for a given research question, what parameter values are biologically or psychologically plausible, how to validate model identifiability, or how to interpret fitted parameters in terms of cognitive mechanisms.

This router skill directs you to specialized sub-skills for:

- **Symbolic/production systems** (ACT-R): Modeling memory retrieval, skill acquisition, problem-solving
- **Evidence accumulation models** (DDM, LBA): Decomposing choice response-time data into drift rate, boundary separation, and non-decision time
- **Bayesian hierarchical models**: Estimating individual differences with partial pooling, prior specification, MCMC diagnostics
- **Spiking neural networks**: Biologically realistic network simulations with neuron models, synaptic dynamics, and plasticity
- **Model validation**: Parameter recovery studies to verify identifiability before interpreting fitted values

## When to Use This Skill

Use this skill when:

- Designing a computational model of a cognitive process (decision-making, memory, learning, perception, attention)
- Choosing between modeling frameworks for a specific research question
- Decomposing behavioral data (RT, accuracy, learning curves) into latent cognitive components
- Estimating parameters that have psychological interpretations (evidence quality, response caution, learning rate, memory activation)
- Validating that your model's parameters are identifiable from your experimental design
- Comparing competing cognitive theories via formal model comparison

Do NOT use this skill for:

- Pure machine learning / predictive modeling without cognitive interpretation (use standard ML frameworks)
- Analyzing neural data without an explicit cognitive model (see `erp-analysis`, `fmri-glm-analysis-guide`)
- Simple descriptive statistics on behavioral data (mean RT, accuracy) without process decomposition

## Research Planning Protocol

Before executing the domain-specific steps below, you MUST:

1. **State the research question** — What cognitive mechanism or process are you trying to understand?
2. **Justify the method choice** — Why computational modeling (not descriptive statistics, not pure ML)? What alternatives were considered?
3. **Declare expected outcomes** — Which parameters do you expect to differ across conditions? What pattern would support vs. refute your hypothesis?
4. **Note assumptions and limitations** — What does your chosen model assume about the cognitive process? Where could it mislead?
5. **Present the plan to the user and WAIT for confirmation** before proceeding.

|

|   |   (Dayan & Abbott, 2001)

### Step 5: Model Validation (ALWAYS REQUIRED)

```
Before interpreting ANY fitted parameters, you MUST validate identifiability:

--> PARAMETER RECOVERY STUDY
    See references/parameter-recovery-checker.md

    Procedure:
    1. Sample 100-1000 ground-truth parameter sets across plausible ranges
    2. Simulate data from each set (matching your experimental design exactly)
    3. Fit the model to simulated data using your exact pipeline
    4. Compare recovered vs. true parameters

    Quality criteria:
    - Pearson r > 0.9 (good), > 0.8 (acceptable), < 0.7 (concerning)
    - Bias < 10% of parameter range
    - Check parameter correlations: |r| > 0.5 indicates tradeoffs

    Common tradeoffs:
    - DDM: drift rate (v) vs. boundary (a)
    - ACT-R: noise (s) vs. threshold (tau)
    - RL: learning rate (alpha) vs. inverse temperature (beta)

    If recovery fails:
    - Fix some parameters to theoretically motivated values
    - Collect more data (increase trial count)
    - Reparameterize the model
    - Report the tradeoff and interpret cautiously
```

## Framework Overviews

### ACT-R (Symbolic Cognitive Architecture)

**Core idea**: Cognition emerges from production rules operating on declarative chunks stored in module buffers. Subsymbolic activation determines retrieval times and probabilities.

**Key parameters**:
- Base-level decay (d): **0.5** (Anderson & Schooler, 1991)
- Activation noise (s): **0.4** (Anderson, 2007)
- Latency factor (F): **1.0** (Anderson, 2007)
- Production cycle time: **50 ms** (Anderson, 2007)

**When to use**: Memory retrieval, skill acquisition, problem-solving, multi-step reasoning tasks where symbolic structure matters.

**Software**: ACT-R 7.x (Lisp), pyactr (Python)

**See**: `references/act-r-model-builder.md`

### Drift-Diffusion Model (Evidence Accumulation)

**Core idea**: Noisy evidence accumulates from a starting point toward one of two decision boundaries. RT = decision time + non-decision time.

**Key parameters**:
- Drift rate (v): **0.5-3.0** — evidence quality (Ratcliff & McKoon, 2008)
- Boundary separation (a): **0.8-2.0** — response caution (Ratcliff & McKoon, 2008)
- Non-decision time (t0): **0.2-0.5 s** — encoding + motor (Ratcliff & McKoon, 2008)
- Starting point (z): **a/2** (unbiased) — response bias

**When to use**: Two-choice RT tasks with speed-accuracy tradeoff; minimum 40-50 trials per condition.

**Software**: HDDM (Python), fast-dm (C/R), PyDDM (Python)

**See**: `references/drift-diffusion-model.md`, `references/evidence-accumulation-selector.md`

### Linear Ballistic Accumulator (Multi-Choice Evidence Accumulation)

**Core idea**: Independent linear accumulators (one per response option) race to a common threshold. First to reach threshold wins.

**Key parameters**:
- Mean drift rate (vi): evidence for option i
- Drift variability (s): often fixed to 1.0 for scaling
- Threshold (b): evidence needed to respond
- Starting point variability (A): uniform [0, A]
- Non-decision time (t0)

**When to use**: Tasks with 3+ response alternatives; minimum ~100 trials per condition.

**Software**: rtdists (R), DMC (R)

**See**: `references/evidence-accumulation-selector.md`

### Bayesian Hierarchical Models

**Core idea**: Individual parameters are drawn from group-level distributions (partial pooling). Priors encode cognitive constraints. MCMC estimates full posterior distributions.

**Key decisions**:
- Pooling strategy: partial pooling (default for cognitive science; Lee & Wagenmakers, 2014)
- Parameterization: non-centered for few observations per group (Betancourt & Girolami, 2015)
- Priors: weakly informative, respecting constraints (RT > 0, rates in [0,1])

**Convergence criteria**:
- R-hat < **1.01** (Vehtari et al., 2021)
- Bulk-ESS > **400**, Tail-ESS > **400**
- Divergent transitions = **0**

**When to use**: Small-to-moderate samples (N=15-40), individual differences, parameter uncertainty matters.

**Software**: Stan, PyMC, HDDM (for DDM specifically)

**See**: `references/bayesian-cognitive-model-builder.md`

### Spiking Neural Networks

**Core idea**: Biologically realistic simulations of neural circuits with explicit spike timing, synaptic dynamics, and plasticity rules.

**Key decisions**:
- Neuron model: LIF (simple), AdEx (adaptation/bursting), Izhikevich (diverse patterns)
- E/I balance: 80/20 ratio, g = 4-8 for asynchronous irregular state (Brunel, 2000)
- Synaptic time constants: AMPA (5 ms), NMDA (100 ms), GABA_A (10 ms)
- Integration time step: **0.1 ms** for LIF/AdEx (Rotter & Diesmann, 1999)

**Validation metrics**:
- Excitatory firing rate: **1-10 Hz** (Brunel, 2000)
- CV of ISI: **~1.0** (irregular firing; Softky & Koch, 1993)
- Pairwise correlation: **0.01-0.1** (Cohen & Kohn, 2011)

**When to use**: Neural implementation questions, spike timing matters, synaptic plasticity, network dynamics.

**Software**: NEST (large-scale), Brian2 (prototyping), NEURON (compartmental), GeNN (GPU)

**See**: `references/spiking-network-model-builder.md`

## Common Pitfalls Across All Frameworks

1. **Fitting without parameter recovery**: Never interpret fitted parameters without first validating that your pipeline can recover known parameters from simulated data (Heathcote et al., 2015; Wilson & Collins, 2019).

2. **Too many free parameters for the data**: Rule of thumb: number of free parameters should be substantially less than the number of independent data points. For DDM: 200+ trials per condition for full 7-parameter model (Ratcliff & Childers, 2015).

3. **Ignoring parameter tradeoffs**: High marginal recovery (good r for each parameter) can coexist with strong cross-parameter correlations that distort interpretation. Always check the parameter correlation matrix.

4. **Analyzing only summary statistics**: Mean RT discards distributional information that cognitive models leverage. Fit full RT distributions (quantiles) for DDM/LBA. Fit trial-by-trial data for RL/ACT-R.

5. **Confusing model fit with model validity**: A model can fit data well and still be wrong. Always compare against plausible alternatives using formal model comparison (BIC, WAIC, Bayes factors).

6. **Not checking convergence (Bayesian models)**: Report R-hat, ESS, and divergent transitions. R-hat < 1.01 is the modern standard (Vehtari et al., 2021), not the outdated < 1.1.

7. **Treating parameters as fixed truths**: Parameters are estimates with uncertainty. Report credible intervals (Bayesian) or confidence intervals (frequentist), not just point estimates.

8. **Skipping posterior predictive checks**: Simulate data from fitted parameters and verify the model reproduces key features of observed data (RT quantiles, accuracy, learning curves).

## Minimum Reporting Checklist

When publishing computational cognitive modeling results:

- [ ] Model specification: full generative model with all parameters defined
- [ ] Justification for model choice: why this framework? What alternatives were considered?
- [ ] Data characteristics: trial counts per condition, sample size, exclusion criteria
- [ ] Free vs. fixed parameters: which parameters were estimated vs. fixed, with rationale
- [ ] Fitting method: MLE, chi-square, Bayesian MCMC (with software and version)
- [ ] Convergence diagnostics (if Bayesian): R-hat, ESS, divergences
- [ ] Parameter recovery results: correlation, bias, RMSE for each parameter
- [ ] Model fit assessment: quantile-probability plots, posterior predictive checks, or equivalent
- [ ] Model comparison (if applicable): BIC/AIC/WAIC/Bayes factors with standard errors
- [ ] Parameter estimates: means/medians with uncertainty (SE, CI, or credible intervals)
- [ ] Interpretation: map parameters to cognitive processes with citations
