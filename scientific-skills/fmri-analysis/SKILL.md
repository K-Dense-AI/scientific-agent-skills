# fMRI Analysis

## Purpose

This skill provides comprehensive guidance for the complete fMRI research pipeline, from initial task design through preprocessing to statistical analysis. Functional MRI (fMRI) measures brain activity by detecting blood oxygenation level-dependent (BOLD) signal changes. Successfully conducting fMRI research requires domain-specific expertise at every stage: designing experiments that maximize statistical power while respecting hemodynamic constraints, preprocessing data to remove artifacts without destroying neural signal, and specifying statistical models that correctly account for the temporal and spatial structure of fMRI data.

A competent programmer without neuroimaging training would make critical errors at each stage: choosing fixed inter-stimulus intervals that waste statistical power, smoothing data before multivariate analysis and destroying spatial patterns, using inappropriate multiple comparison corrections that inflate false positives, or specifying GLMs with collinear regressors that make contrasts uninterpretable.

This skill consolidates three interconnected sub-skills covering the full fMRI analysis pipeline. Each stage builds on the previous one, and decisions made early in the pipeline constrain options later.

## When to Use This Skill

Use this skill when:

- Setting up preprocessing pipelines for fMRI data

## Research Planning Protocol

Before executing the domain-specific steps below, you MUST:

1. **State the research question** — What specific brain-behavior hypothesis is this study testing?
2. **Justify the method choice** — Why fMRI? What alternatives (EEG, MEG, behavior-only, lesion studies) were considered and why were they rejected?
3. **Declare expected outcomes** — Which brain regions do you expect to activate? What direction of effects? What would falsify your hypothesis?
4. **Note assumptions and limitations** — What does fMRI assume about neurovascular coupling? What cognitive processes might your task confound? What populations or brain regions are problematic for fMRI?
5. **Present the plan to the user and WAIT for confirmation** before proceeding.

## Pipeline Overview

The fMRI analysis pipeline consists of three major stages, each with critical decision points:

```
Stage 1: Task Design
├── Choose design type (block / event-related / mixed)
├── Optimize timing parameters (ISI, jittering, trial counts)
├── Calculate statistical power and efficiency
└── Integrate behavioral constraints with BOLD timing

Stage 2: Preprocessing
├── Convert DICOM to NIfTI (BIDS format)
├── Motion correction and quality control
├── Distortion correction (fieldmap-based)
├── Spatial normalization to template space
├── Smoothing (conditional on analysis type)
└── Extract confound time series

Stage 3: Statistical Analysis (GLM)
├── Specify HRF model and design matrix
├── Configure temporal filtering and confound regression
├── Define contrasts to test hypotheses
├── Perform group-level inference
└── Apply multiple comparison correction
```

Each stage is detailed in a dedicated reference file. The sections below provide routing logic to help you navigate to the appropriate sub-skill.

## Stage 1: Task Design

**Reference file**: `references/fmri-task-design.md`

### When to Consult This Reference

Consult the task design reference when:

- Planning a new fMRI experiment before data collection
- Choosing between block, event-related, or mixed designs
- Determining optimal inter-stimulus intervals and jittering strategies
- Calculating minimum trial counts per condition
- Optimizing design efficiency for specific contrasts
- Balancing behavioral task requirements with BOLD timing constraints

### Key Decision Points

The task design reference will guide you through:

1. **Design type selection** — Block designs maximize detection power (Friston et al., 1999) but cannot estimate trial-by-trial responses. Event-related designs enable flexible trial-level analysis (Dale, 1999) but require more trials. Mixed designs separate sustained and transient effects (Petersen & Dubis, 2012)

2. **Timing optimization** — The BOLD response peaks at 4-6 seconds and returns to baseline after 12-16 seconds (Glover, 1999). Jittered inter-stimulus intervals dramatically improve design efficiency compared to fixed timing (Dale, 1999)

3. **Statistical power** — Minimum trial counts depend on design type: event-related designs require at least 20-30 trials per condition (Desmond & Glover, 2002), while block designs need 4-6 blocks of 15-20 seconds each (Bandettini et al., 1993)

### Cross-References to Later Stages

- Task timing parameters (ISI, block duration) determine the appropriate high-pass filter cutoff in GLM analysis (Stage 3)
- Design type (block vs. event-related) influences HRF model selection (canonical vs. FIR) in GLM specification
- Trial counts affect whether multivariate analyses (MVPA) are feasible, which determines smoothing decisions in preprocessing (Stage 2)

## Stage 2: Preprocessing

**Reference file**: `references/fmri-preprocessing-pipeline.md`

### When to Consult This Reference

Consult the preprocessing reference when:

- Setting up a preprocessing pipeline for newly acquired fMRI data
- Choosing between preprocessing tools (fMRIPrep, FSL, SPM, AFNI)
- Deciding which preprocessing steps to include or skip
- Selecting parameters for motion correction, smoothing, or normalization
- Performing quality control on preprocessed data
- Troubleshooting preprocessing failures or artifacts

### Key Decision Points

The preprocessing reference will guide you through:

1. **Pipeline order** — The standard sequence is: discard non-steady-state volumes → slice timing correction (conditional) → motion correction → distortion correction → coregistration → normalization → smoothing (conditional). Deviations from this order can introduce artifacts

2. **Slice timing correction** — Apply for TR > 1 second; skip for multiband acquisitions with TR < 1 second (Sladky et al., 2011; HCP consortium)

3. **Motion correction** — Use 6-parameter rigid body alignment (Jenkinson et al., 2002). Save motion parameters for confound regression in Stage 3

4. **Smoothing** — Kernel size depends on analysis type: 6-8 mm FWHM for univariate GLM (Poldrack et al., 2011), 4-6 mm for resting-state, NONE for MVPA (Misaki et al., 2013). This is the most common preprocessing error

5. **Tool selection** — fMRIPrep (Esteban et al., 2019) is recommended as the default for most studies. It provides automated, analysis-agnostic preprocessing with comprehensive quality control reports

### Cross-References to Other Stages

- Motion parameters extracted during preprocessing are used as confound regressors in GLM analysis (Stage 3)
- Smoothing decisions depend on the analysis type planned in task design (Stage 1): MVPA requires no smoothing
- Distortion correction quality affects the accuracy of spatial localization in final results
- The high-pass filter cutoff in GLM analysis (Stage 3) must be compatible with the task design timing (Stage 1)

## Stage 3: Statistical Analysis (GLM)

**Reference file**: `references/fmri-glm-analysis.md`

### When to Consult This Reference

Consult the GLM analysis reference when:

- Specifying a first-level (single-subject) GLM for task fMRI
- Choosing HRF models, confound regressors, and temporal filtering parameters
- Defining contrasts to test experimental hypotheses
- Setting up second-level (group) analyses
- Selecting appropriate multiple comparison correction methods
- Reviewing or troubleshooting statistical results

### Key Decision Points

The GLM analysis reference will guide you through:

1. **HRF model selection** — Canonical double-gamma HRF is the default (Lindquist et al., 2009). Add temporal derivatives when peak latency may vary (Friston et al., 1998). Use FIR models when HRF shape is unknown but you have many trials (>40 per condition; Glover, 1999)

2. **High-pass filter cutoff** — Default is 128 seconds (Poldrack et al., 2011). The cutoff period must be at least 2x the longest interval between events of the same condition type, or you will attenuate your experimental signal

3. **Confound regression** — At minimum, include 6 motion parameters. The 24-parameter model (Friston et al., 1996) is recommended. Add aCompCor components from white matter and CSF (Behzadi et al., 2007). Include spike regressors for volumes with framewise displacement > 0.5 mm (Power et al., 2012)

4. **Multiple comparison correction** — Cluster-based inference with cluster-defining threshold p < 0.001 is standard (Eklund et al., 2016). Never use p < 0.01 as CDT (inflates false positives to 70%). Permutation testing with TFCE is the gold standard (Smith & Nichols, 2009)

5. **Group inference** — Use mixed-effects or OLS summary statistics for group-level inference (Mumford & Nichols, 2009). Never use fixed-effects models for group inference; results would not generalize beyond the scanned subjects

### Cross-References to Earlier Stages

- High-pass filter cutoff depends on task timing from Stage 1 (must preserve the fundamental frequency of your experimental design)
- Motion parameters and other confounds come from Stage 2 preprocessing outputs
- Smoothing kernel applied in Stage 2 affects the validity of Random Field Theory corrections
- HRF model choice may depend on design type from Stage 1 (FIR requires many trials)

## Integration Across Stages

### Forward Dependencies (Early Decisions Constrain Later Options)

1. **Task design → Preprocessing**:
   - MVPA designs require no smoothing or minimal smoothing
   - Rapid event-related designs with short TR may skip slice timing correction
   - Trial counts determine whether complex HRF models are feasible

2. **Task design → GLM**:
   - Block duration and ISI determine high-pass filter cutoff
   - Design type (block vs. event-related) influences HRF model selection
   - Jittering strategy affects design matrix efficiency

3. **Preprocessing → GLM**:
   - Motion parameters become confound regressors
   - Smoothing kernel affects spatial extent thresholds
   - Temporal resolution (TR) determines autocorrelation model

### Backward Dependencies (Later Analyses Inform Earlier Choices)

1. **Planned analysis → Task design**:
   - MVPA requires more trials per condition (40+) than univariate GLM (20+)
   - ROI analyses require fewer trials than whole-brain searchlight
   - Connectivity analyses require longer scan durations

2. **Planned analysis → Preprocessing**:
   - MVPA: skip smoothing entirely
   - Univariate GLM: smooth at 2-3x voxel size
   - Surface-based analysis: use surface normalization instead of volumetric

## Common Cross-Stage Pitfalls

1. **Smoothing before MVPA** — Destroys the fine-grained spatial patterns that multivariate methods rely on. This error occurs when preprocessing is performed without considering the planned analysis type (Misaki et al., 2013)

2. **High-pass filter too aggressive for slow designs** — If your task has long inter-trial intervals (e.g., 90 seconds), the default 128-second filter cutoff will attenuate your signal. The filter cutoff must be determined by task timing, not by software defaults

3. **Insufficient trials for complex HRF models** — FIR models require 40+ trials per condition for stable estimation (Glover, 1999). Using FIR with fewer trials produces unreliable estimates

4. **Motion confounds not properly modeled** — Motion parameters extracted during preprocessing must be included in the GLM. Omitting them is the single largest source of false positives in fMRI (Power et al., 2012)

5. **Design efficiency not optimized** — Using fixed ISI instead of jittered timing can reduce statistical power by 10x or more (Dale, 1999). This is a task design error that cannot be corrected during analysis

6. **Cluster-defining threshold too liberal** — Using p < 0.01 as the cluster-defining threshold produces false positive rates up to 70% instead of the nominal 5% (Eklund et al., 2016). Always use p < 0.001 or stricter
